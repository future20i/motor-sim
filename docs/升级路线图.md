## v2.0 — 评分体系实战化（优先做，改动最小）

### 为什么先做这个

v2.0 不动任何仿真逻辑，只改评分和分析——改动范围最小（~200行），但能让结论从「学术排名」变成「工程决策依据」。

### 2.1 硬约束一票否决

```python
HARD_CONSTRAINTS = {
    "vdc_overvoltage":  lambda m: m.vdc_max_dev < 20.0,    # 母线过压 <20%
    "flywheel_stall":    lambda m: m.soc_min > 0.05,        # SOC 不低于 5%
    "overload":          lambda m: m.p_flywheel_max < 1.5,  # 飞轮不过载 150%
}
```

任何策略触发硬约束 → 标记为「不合格」，不参与排名。这和真实产品验收逻辑一致——IEC 62040 和 GB/T 标准对 UPS 都有明确的安全边界。

### 2.2 帕累托前沿可视化

不求和，直接画出每个策略在多维空间的位置：

```
    频率响应分 ↑
               │
   AI策略 ●    │      ← 频率响应好，Vdc 也好
               │
               │      ● 固定VSG  ← Vdc 不错，频率响应一般
               │
               │            ● 固定VDC  ← Vdc 好，频率响应差
               └────────────────→ Vdc 稳压分

被支配区 → 在 Vdc 和频率双维度上都不如别的策略 → 可以淘汰
帕累托前沿 → 不存在在所有维度上都优于它的策略 → 候选策略池
```

输出一个明确的决策表：

| 策略 | Vdc分 | 频率分 | SOC分 | 是否在帕累托前沿 | 劣势维度 |
|---|---|---|---|---|---|
| AI自适应 | 92 | 85 | 88 | ✅ | — |
| AI规则引擎 | 91 | 83 | 87 | ✅ | 频率(-2 vs 自适应) |
| 固定VSG | 75 | 60 | 80 | ❌ | 被AI自适应支配 |
| 固定VDC | 95 | 30 | 90 | ✅ | 频率(-55 vs AI自适应) |

这样你选策略时能精确知道：**固定VDC 在纯 Vdc 稳压场景最强，但频率事件来了完全不行——如果你已知客户的电网频率稳定，它是理性选择。**

### 2.3 蒙特卡洛鲁棒性

同样策略、同样场景，但场景参数随机抖动 ±20%，跑 100 次：

```
策略         均值    标准差   最低分   5%分位数
─────────────────────────────────────────────
AI自适应     76.4   ±3.2     68.1     71.2
AI规则引擎   75.9   ±8.7     52.3     61.0
固定VSG      46.0   ±15.1    22.5     28.7
```

**均值高 + 方差低 = 真正可靠**。`5% 分位数` 是你对客户承诺的保守边界——「在最差的 5% 情况下，这个策略仍然能拿到 71 分」。

## v2.5 — AI 真正变 AI（强化学习）

### 为什么当前不是 AI

```python
# s4_ai_controller.py 的实际逻辑
def _rule_based(self, snap):
    if abs(snap.df) > 0.2:
        return Action(mode=FREQ_REGULATION, droop=0.04)
    elif abs(snap.dVdc) > 30:
        return Action(mode=DC_BUS_CONTROL)
    else:
        return Action(mode=IDLE)
```

这是人写的规则树——**阈值是人设的、逻辑是人定的、没有任何学习过程**。

### RL 改造方案

把 motor-sim 包装成标准 Gym 环境，然后用 Stable-Baselines3 的 PPO 或 SAC 算法训练：

```python
import gymnasium as gym
from stable_baselines3 import PPO

class FlywheelEnv(gym.Env):
    """
    状态空间: [f, df, dfdt, Vdc, dVdc, SOC, P_grid, P_load, f_pll, Vg_amplitude]
    动作空间: 离散(5种mode) × 连续(droop, deadband, dc_mode)
    奖励: Vdc分 + 频率分 + SOC分 - 越限惩罚
    """
    
    def step(self, action):
        mode, droop, deadband, dc_mode = self._decode(action)
        
        # 步进一个控制周期
        orchestrator.mode = mode
        orchestrator.freq_reg.droop = droop
        orchestrator.dc_ctrl.mode = dc_mode
        
        # 跑仿真
        grid.step()
        inverter.step(...)
        dc_bus.step(...)
        flywheel.step(...)
        
        # 计算奖励
        reward = self._compute_reward()
        done = self._is_terminal()
        
        return observation, reward, done, info

# 训练
env = FlywheelEnv()
model = PPO("MlpPolicy", env, verbose=1)
model.learn(total_timesteps=1_000_000)

# 导出策略 → 替换 s4_ai_controller.py
model.save("flywheel_rl_policy")
```

### 为什么 RL 适合飞轮控制

飞轮 UPS 的控制是典型的**序列决策**问题：

- 你现在放电稳频 → SOC 下降 → 下一波扰动来了扛不住
- 你现在充电补 SOC → 当前频率偏差不管 → 客户不满

启发式规则只能看当前一拍。RL 网络可以学到**跨时间的权衡**——「这个扰动不大，先放 30% 功率扛过去，留 70% 给后面可能的大扰动」。

而且 RL 不需要人设定死区阈值——网络自己在训练中学会「什么程度的偏差才值得出手」。

### 可解释性（为什么敢用在产品上）

RL 的致命弱点是黑盒。实战中必须加可解释性层：

```python
# 用 SHAP 分析每个输入特征对决策的贡献
import shap
explainer = shap.Explainer(model.policy)
shap_values = explainer(observation)

# 输出:
# df       → 45% 贡献  (频率偏差是最大决策因子)
# dVdc     → 28% 贡献
# SOC      → 18% 贡献
# P_load   →  6% 贡献
# ...      →  3% 贡献
```

同时记录完整的决策日志：

```json
{
  "t": 2.350,
  "obs": {"df": 0.42, "dVdc": 15, "SOC": 0.45},
  "action": "FREQ_REGULATION",
  "droop": 0.038,
  "reason": "df>0.3 && SOC>0.3 → 调频模式",
  "confidence": 0.92
}
```

这样即使出了 RL 策略，每次决策都有据可查——出问题时能回溯「为什么当时选了那个动作」。

## v3.0 — 场景生成实战化

### 3.1 对抗场景生成（自动找弱点）

当前 10 个场景是人写的，最危险的 corner case 可能是人想不到的组合：**频率骤降 + 同时电压暂降 + 半小时前 SOC 已经偏低**。

对抗生成器的逻辑：

```python
from scipy.optimize import differential_evolution

def adversarial_score(params):
    """这个场景参数组合下，策略的得分（越低越好，我们要找最低点）"""
    scenario = ScenarioConfig(
        freq_dip_magnitude=params[0],    # 频率骤降幅度
        v_sag_magnitude=params[1],       # 电压暂降幅度
        initial_soc=params[2],           # 初始SOC
        duration=params[3],              # 持续时间
    )
    runner = S4ExperimentRunner()
    result = runner.run_scenario(scenario, strategy=target_strategy)
    return result.metrics.composite_score  # 我们要最小化这个

# 自动搜索最恶意的场景参数组合
result = differential_evolution(
    adversarial_score,
    bounds=[(0.2, 2.0), (0.3, 0.9), (0.05, 0.8), (0.1, 10.0)],
)
print(f"AI自适应最弱场景: {result.x} → 得分 {result.fun:.1f}")
```

这等于让算法自己找到「AI自适应在什么情况下会翻车」，然后你可以决定——这个情况在真实电网中会不会发生？如果会，能不能接受？

### 3.2 真实录波回放

泓慧如果有台架测试或客户现场录到的真实电网波形（.csv/.mat/.dat），直接导入替代合成波形：

```python
class GridRecorder:
    """从录波文件播放真实三相电压/电流"""
    def __init__(self, filepath):
        self.data = pd.read_csv(filepath)  # Va,Vb,Vc,t
        self.idx = 0
    
    def step(self):
        row = self.data.iloc[self.idx]
        self.idx += 1
        return row['Va'], row['Vb'], row['Vc']
```

这样跑出来的 S4 结果不再是「仿真环境下的排名」，而是「在你们客户真实电网环境下的排名」——对客户的说服力直接升一个量级。

## v3.5 — 多机 + HIL

### 多飞轮协调

真实数据中心通常 4-8 台飞轮并联。多机带来了新问题——SOC 均衡。

```python
class FlywheelFarm:
    """多飞轮集群"""
    def __init__(self, n=4):
        self.units = [FlywheelEnergyStorage() for _ in range(n)]
    
    def dispatch(self, total_P_ref):
        """SOC 高的多出力，SOC 低的多充电 → 自动均衡"""
        socs = [u.state.SOC for u in self.units]
        weights = [s / sum(socs) for s in socs]
        for i, u in enumerate(self.units):
            u.set_power(target=total_P_ref * weights[i])
```

多机均衡是一个很好的 RL 训练目标——网络不仅学会「什么时候出力」，还要学会「哪台飞轮出力」。

### HIL (硬件在环)

`virtual_opcua_server.py` 已经可以把仿真暴露成 OPC-UA 服务器。打通之后：

```
真实 PLC/控制器 ←→ OPC-UA ←→ motor-sim 仿真

```

这让你们可以在**有真实飞轮硬件之前**，用仿真环境测试真实的 PLC 控制代码——SIL（软件在环）→ HIL（硬件在环）。
